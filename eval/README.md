# GrowthMate 评测体系

这套评测不属于用户前台，而是产品迭代用的离线实验台。目标是回答一个问题：

> Prompt、Agent 流程或模型改动之后，GrowthMate 的输出有没有真的变好？

## 当前版本

第一版采用“固定测试集 + 固定 Rubric + 启发式自动评分 + 人工复核”的方式。

- `rubric.json`：完整评分标准，总分 100。
- `cases.json`：第一批典型 KOC 测试画像、发布数据和真实发布稿。
- `platform-plan.md`：评测后台的平台化方案，用于后续可视化、人工标注和 Prompt 版本对比。
- `reports/`：运行 `npm run eval:live` 后生成 JSON 和 Markdown 报告。

## 评测框架

这套标准按 Agent 完整链路拆成 5 层，尽量做到 MECE：每一层只回答一个问题，避免重复打分。

| 层级 | 权重 | 核心问题 | 对应维度 |
| --- | ---: | --- | --- |
| 输入理解层 | 15 | 是不是给这个用户的方案？ | 用户画像贴合度 |
| 策略决策层 | 30 | 为什么现在应该这样做？ | Agent 工作流完整度、增长任务设计 |
| 执行交付层 | 40 | 用户能不能照着做？ | 具体可执行性、平台适配、内容可用性 |
| 反馈迭代层 | 10 | 发完以后怎么改？ | 复盘质量 |
| 信任边界层 | 5 | 这个建议安全吗？ | 边界与可信度 |

面试时可以这样讲：

> 我没有只看文案好不好，而是把 KOC Agent 的能力拆成输入理解、策略决策、执行交付、反馈迭代和风险边界五层。这样既能评价生成内容本身，也能评价 Agent 是否真的形成增长闭环。

## 人工标注方法

每个二级维度使用 1-5 分：

- 5 分：高度贴合用户输入，可执行、可解释，并能支撑下一轮迭代。
- 4 分：整体可用，有少量泛化或细节不足。
- 3 分：能完成基本任务，但缺少个性化判断或执行细节。
- 2 分：输出偏模板化，用户难以直接使用。
- 1 分：偏离用户画像、结构缺失或存在明显误导。

标注时不要只看“写得好不好”，而要看证据：

- 是否引用了用户真实输入，例如身份、素材、平台、卡点。
- 是否给出动作、顺序、素材和观察指标。
- 是否能解释为什么先做这条，而不是随机生成标题。
- 是否把复盘结论和真实数据、评论、发布稿对应起来。
- 是否出现虚假能力或过度承诺。

`rubric.json` 里每个维度都包含 `criteria`、`annotationGuide`、`positiveExample` 和 `negativeExample`，后续人工标注可以直接按这些字段做表单。

## 使用方式

前台联动：

```text
http://localhost:5176/
```

用户每完成一次方案生成、内容生成或复盘，前台会自动调用 `/api/eval/snapshot` 写入评测快照。

评测后台：

```text
http://localhost:5176/eval
```

后台会展示最近评测、平均分、五层框架得分和各维度明细。

只校验评测标准和测试集结构：

```bash
npm run eval
```

调用本地 Agent API 跑真实评测：

```bash
npm run dev
npm run eval:live
```

评测后台默认只展示真实模型评测报告。不要用 mock/demo 数据冒充 benchmark 分数；如果模型超时，应在后台展示失败报告或暂无真实报告。

如果服务端口不是默认端口，可以指定：

```bash
node scripts/run-eval.js --live --base-url=http://127.0.0.1:5176
```

调试单个用例：

```bash
node scripts/run-eval.js --live --case=workplace_newcomer_xhs
```

设置单个 API 请求超时时间：

```bash
node scripts/run-eval.js --live --timeout-ms=180000
```

## 如何看报告

报告会生成在 `eval/reports/` 下：

- `.json`：保留完整结构，方便后续做趋势对比。
- `.md`：方便人工阅读和放进项目说明。

当前自动分数是启发式的，不等于最终质量判断。每次 Prompt 大改后，建议看三类信息：

- 总分是否明显变化。
- 哪个维度下降，例如平台适配、复盘质量、内容可用性。
- 是否命中 `avoid` 词，例如保证涨粉、抓取真实数据。
